# 功能抽离、封装与发布工作流（中文说明）

## 1. 目的

这份文档说明一套通用工作流，用于把项目中的某一段核心功能从主脚本中抽离出来，放进一个独立文件夹，补齐可安装配置，让其他人可以：

- 直接 `import` 调用
- 直接命令行运行
- 直接 `pip install -e ...` 做可编辑安装

这套方法不局限于本次的 `TOPPRA + MVC + Ruckig` 功能，也适用于其他需要“从实验脚本演变为可复用模块”的场景。

---

## 2. 什么时候应该抽离成独立文件夹

建议在下面几种情况下做抽离：

- 主脚本已经过长，逻辑耦合严重
- 同一段功能需要被多个脚本重复调用
- 需要对外提供给其他同事或其他项目使用
- 需要命令行入口，而不只是内部函数
- 需要固定一套默认参数、输入输出约定和交付格式

如果只是临时验证，不需要立即抽离；如果功能已经稳定且会复用，就应该尽早从主脚本里拆出来。

## 4. 通用封装思路

建议把“实验脚本”拆成下面三层：

### 4.1 核心实现层

职责：

- 只做算法与数据处理
- 不依赖特定画图流程
- 不依赖特定主脚本的临时变量

典型文件：

- `pipeline.py`

适合放进去的内容：

- 数据加载
- 路径预处理
- 约束构建
- 求解器调用
- 结果拼接
- 结果结构定义

### 4.2 精简 API 层

职责：

- 为调用方提供更干净的入口
- 屏蔽内部细节配置

典型文件：

- `core.py`

适合放进去的内容：

- `CorePipelineConfig`
- `run_core_pipeline(...)`

这样做的目的是：内部实现可以复杂，但别人调用时不需要理解所有底层细节。

### 4.3 命令行入口层

职责：

- 允许别人不写 Python 代码，直接运行
- 固定输入输出格式
- 把结果导出为文件

典型文件：

- `cli.py`
- `__main__.py`

这样别人可以直接执行：

```bash
python -m your_package
```

或者安装后执行：

```bash
your-console-script
```

## 5. 推荐目录结构

推荐的最小可交付结构如下：

```text
your_package/
├─ pipeline.py
├─ core.py
├─ cli.py
├─ example_usage.py
├─ __init__.py
├─ __main__.py
├─ README.md
├─ 封装发布工作流.md
├─ requirements.txt
└─ pyproject.toml
```

各文件职责：

- `pipeline.py`：完整实现
- `core.py`：精简 API
- `cli.py`：命令行入口
- `example_usage.py`：最小示例
- `__init__.py`：对外导出公共接口
- `__main__.py`：支持 `python -m`
- `README.md`：英文或简洁说明
- `封装发布工作流.md`：中文交付说明
- `requirements.txt`：依赖列表
- `pyproject.toml`：安装与发布配置

## 6. 抽离功能时的实际步骤

### 6.1 先明确输入和输出

在动手拆代码之前，先写清楚：

- 输入是什么
- 输出是什么
- 哪些参数应该暴露给外部
- 哪些参数只应该保留在内部

如果这一步不做，抽离后的包很容易继续带着原脚本的隐含耦合。

例如本次功能的主要输入输出就是：

- 输入：
  - 关节路径文件
  - 机器人模型
  - 关节速度/加速度限制
  - Ruckig 参数
- 输出：
  - `qs / qds / qdds`
  - `mvc_upper`
  - `minima_idx`
  - `segment_targets`
  - `segment_target_velocities`

### 6.2 切断主脚本依赖

抽离时要特别注意下面几类坏味道：

- 直接读取主脚本的全局变量
- 直接依赖主脚本的 `args`
- 直接在核心函数里 `plt.show()`
- 核心逻辑和导出逻辑写在一起

正确做法是：

- 核心逻辑只返回数据
- 导出和画图在外层脚本处理
- 参数全部由配置对象或函数参数传入

### 6.3 用配置对象固定行为

建议用 `dataclass` 存配置，而不是散落一堆位置参数。

例如：

- `PathPreparationConfig`
- `ToppraConfig`
- `RuckigConfig`
- `CorePipelineConfig`

好处：

- 参数含义明确
- 默认值集中
- 后续改默认值时不容易漏改

### 6.4 把结果也结构化

不要只返回裸 `tuple`，建议定义结果对象，例如：

- `PipelineResult`
- `CorePipelineResult`

好处：

- 调用方更容易理解字段含义
- 后续加字段不容易破坏兼容性

## 7. 为什么要补 `pyproject.toml`

如果只有源码目录，没有 `pyproject.toml`，别人通常只能：

- 手动改 `sys.path`
- 或者把目录拷到项目里硬引用

这不属于可靠交付。

补上 `pyproject.toml` 后，别人可以：

```bash
pip install -e fixed_toppra_mvc_ruckig
```

然后直接：

```python
import fixed_toppra_mvc_ruckig
```

或者：

```bash
fixed-toppra-mvc-ruckig --help
```

这就是“可安装交付”和“脚本复制”之间的差别。

## 8. `pyproject.toml` 的主要工作

在本次目录里，文件位置是：

- [pyproject.toml](/C:/Users/hit/PycharmProjects/pythonProject/TOPP-离散2/fixed_toppra_mvc_ruckig/pyproject.toml)

它主要完成四件事：

### 8.1 声明构建系统

```toml
[build-system]
requires = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"
```

作用：

- 告诉 `pip` 用什么工具构建

### 8.2 声明包元信息

```toml
[project]
name = "fixed-toppra-mvc-ruckig"
version = "0.1.0"
requires-python = ">=3.10"
```

作用：

- 告诉安装器包名、版本、Python 版本要求

### 8.3 声明依赖

```toml
dependencies = [
  "numpy>=1.23,<2.0",
  "pandas>=1.5",
  ...
]
```

作用：

- 别人安装时自动检查依赖

### 8.4 声明命令行脚本

```toml
[project.scripts]
fixed-toppra-mvc-ruckig = "fixed_toppra_mvc_ruckig.cli:main"
```

作用：

- 安装后自动生成 `fixed-toppra-mvc-ruckig` 命令

## 9. 当前目录结构下 `pyproject.toml` 的一个关键点

本次包不是这种结构：

```text
repo_root/
└─ fixed_toppra_mvc_ruckig/
   ├─ __init__.py
   └─ ...
```

而是“目录本身就是包目录”：

```text
fixed_toppra_mvc_ruckig/
├─ __init__.py
├─ pipeline.py
├─ core.py
└─ ...
```

所以这里必须显式告诉 `setuptools`：

```toml
[tool.setuptools]
packages = ["fixed_toppra_mvc_ruckig"]

[tool.setuptools.package-dir]
fixed_toppra_mvc_ruckig = "."
```

如果不写这段，`pip install -e` 很容易找不到包。

这属于这种目录布局下最容易漏掉的点。

## 12. 本次抽离中做过的关键修复

除了结构封装，本次还把 Ruckig 相关的稳定性问题一起收进了新包：

### 12.1 禁止负速度

做法：

- `min_velocity = 0.0`
- 输入/输出速度都做非负裁剪

目的：

- 路径参数 `s` 不允许倒退

### 12.2 保留减速能力

做法：

- `min_acceleration = -a_max`

目的：

- 不会因为把下界错误设成 `0` 而失去刹车能力

### 12.3 短段可行性修复

做法：

- 拼接速度不仅看 MVC 最低点，还要看该段长度能否支持
- 对短段做保守速度上限

目的：

- 避免 `RuckigError`：段太短但目标速度太高

### 12.4 自动降级重试

做法：

- 目标速度不行时，自动尝试更软的目标速度

目的：

- 避免某一段直接导致整条轨迹失败

### 12.5 终点状态置零

做法：

- 最终 `sd/sdd = 0`
- 最终 `qd/qdd = 0`

目的：

- 让输出轨迹末端边界更明确

这些修复本身也属于“抽离工作的一部分”，因为别人能否直接调用，不只是目录结构问题，还包括稳定性问题。

## 13. 推荐的交付检查清单

以后再抽离其他功能时，建议按下面清单检查：

### 代码结构

- 核心逻辑是否已经从主脚本剥离
- 是否有 `pipeline.py`
- 是否有精简 API
- 是否有 CLI

### 安装结构

- 是否有 `pyproject.toml`
- 是否有 `requirements.txt`
- 是否能 `pip install -e`

### 对外接口

- 是否有 `__init__.py`
- 是否有 `__main__.py`
- 是否有 console script

### 文档

- 是否写清楚输入
- 是否写清楚输出
- 是否写清楚安装方式
- 是否写清楚运行命令

### 验证

- 是否做过 `py_compile`
- 是否做过 `import`
- 是否做过 `--help`
- 是否做过至少一次冒烟运行

## 15. 总结

把功能“抽离封装到新文件夹并让别人直接调用”，真正需要完成的是四件事：

1. 把核心逻辑从主脚本剥离
2. 给出稳定的 Python API 和 CLI
3. 补齐 `pyproject.toml` 让它可安装
4. 补齐文档和验证，让别人真的能用

目录拆出来只是第一步。  
只有当安装、导入、命令行、冒烟运行都通过之后，这份封装才算真正完成。
